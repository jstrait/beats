BEATS

is a drum machine written in pure Ruby. Feed it a song notated in YAML, and it will produce a precision-milled Wave file of impeccable timing and feel. Here is an example song:

Song:
  Tempo: 120
  Structure:
    - Verse:   x4
    - Chorus:  x2
    - Verse:   x4
    - Chorus:  x2

Verse:
  - bass.wav:      X.......X.......
  - snare.wav:     ....X.......X...
  - hh_closed.wav: X.X.X.X.X.X.X...
  - hh_open.wav:   ..............X.
  - ride.wav:      X...............

Chorus:
  - bass.wav:      X...X...XX..X.XX
  - snare.wav:     ....X.......X...
  - hh_closed.wav: XXXXXXXXXXXXXXXX
  - ride.wav:      ........X.......

And here is what it sounds like after getting the Beats treatment: beat.wav. What a glorious groove!

Installation

BEATS is packaged as a Ruby gem, hosted at gemcutter.org. To install BEATS, run the following command: 

sudo gem install beats

The BEATS source code is hosted at GitHub: http://github.com/jstrait/beats

How to Run

BEATS runs from the command-line. The syntax:

beats [options] INPUT [OUTPUT]

INPUT should be the YAML file for your song. OUTPUT should be the *.wav file where BEATS will place the generated sound. If omitted, it will default to INPUT but with a .wav extension. (For example, the sound for my_song.txt would be saved to my_song.wav).

BEATS supports two command-line options, which can be used together or separately:

-s, --split
    Save each track to a separate Wave file,
    instead of combining them into one.


-p, --pattern
    Only generate a single pattern (such as a
    verse or chorus), instead of the entire song.

Now read on for more about the BEATS file format.

Creating Songs

Songs are notated using YAML, so each song file is just a plain text file you can edit in TextEdit, NotePad, or whatever. The extension doesn't matter.

Each song is divided into one or more patterns. A pattern represents a section of the song, such as a verse, chorus, or bridge. Each pattern in turn has one or more tracks. Each track represents an individual sound, such as a bass drum, snare, or hi-hat.

Songs should start with a header indicating the tempo and the order that the patterns should be played. Patterns can optionally be repeated one or more times. For example, adding "x2" after the pattern name means to play it twice. Since this is YAML, lines must be indented correctly. Also, each pattern in the structure should be preceded by a dash. Here is an example header:

Song:
  Tempo: 120
  Structure:
    - Verse:   x4
    - Chorus:  x2
    - Bridge:  x1
    - Verse:   x4
    - Chorus:  x2

One or more patterns should follow the header. (If there are no patterns, there is nothing to play!) Each pattern definition starts with the pattern name, followed by the rhythm for each track on separate lines.

Verse:
  - bass.wav:      X.......X.......
  - snare.wav:     ....X.......X...
  - hh_closed.wav: X.X.X.X.X.X.X...
  - hh_open.wav:   ..............X.
  - ride.wav:      X...............

Each track line starts with the sound file to use (only 8 or 16-bit Wave files are supported). You can either use a relative or absolute path. Relative paths are from the path of the song file. Track lines must be indented properly and start with a dash.

The track rhythm follows the sound name. An X means that the sound should be triggered, and a . indicates a "rest" between triggers. Each X or . represents a 16th note. You only need to indicate the downbeat of each sound, since sounds will play their full duration (unless cut off by another trigger). For example, to play a quarter note rhythm, X...X...X...X... will do the trick.

As a final note, patterns and the song header can come in any order. If you want to include the song header at the end of the file, you can. You can also mix up the order that patterns are defined.
